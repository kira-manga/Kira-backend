# kira-backend — Full-Project Adversarial Review (2026-07-12)

Four independent review agents (R1–R4) swept the entire codebase after all 10 implementation
phases completed. **Findings-report only — no fixes have been applied.** Each finding below is
verbatim from its reviewer; nothing was filtered out.

**Project state at review time:** `main` @ `baaf1f2` (Phase 10). 232 tests / 0 failures / 53
suites, clean tree. 130 main Kotlin files (9,603 lines), 58 test files (5,648 lines), 266 SQL
lines. Commit chain `32cbe33` (plan) → `8281f8d` … `baaf1f2` (P1–P10), all authored
`Apdelrahman1911 <abdofahmymr14@gmail.com>`, no AI attribution, nothing pushed.

**Review slices:**
- **R1** — security/auth: JWT, login, throttle, user admin, security matrix, exception handling.
- **R2** — source-config core: config model, validation rules 1–33, CanonicalJson (kcj-1), parser modes.
- **R3** — lifecycle/persistence/concurrency: state machine, publication/rollback/import orchestration, locks, migrations V1–V4, startup validators, concurrency ITs.
- **R4** — completion feature, public document/sources API (ETag), DI/config, build, remaining ITs.

## Verdict

**0 CRITICAL. 2 HIGH. 10 MEDIUM. 18 LOW.** The architecture held up under adversarial review:
lock discipline, the 10-step publication transaction, revision monotonicity, import atomicity,
the three-transaction completion pattern, kcj-1 canonicalization, JWT hardening, and ETag
semantics were all independently verified sound. The two HIGHs are small, well-localized
defects — one security-relevant (validation bypass), one robustness (500 instead of 400).

## HIGH

- **[R2-H1] `sourceconfig` validation — Rule 32 header-name check bypassed by whitespace padding.**
  `SourceRules.kt:83-104` — header names are compared against the forbidden/sensitive sets
  without trimming, so `"Authorization "` / `"Cookie "` (trailing space) escape the check and a
  real credential value can be authored into a source config and **published into the public
  document**. Needs `name.trim().lowercase()` before set membership plus RFC 7230 token
  validation (reject names containing whitespace outright).

- **[R4-H1] Completion — client-supplied `model` never length-validated → 500.**
  `CompletionController.kt:47` vs `V5__completions.sql:20` (`model varchar(128)`) — POST
  `/api/v1/completions` with a 129+-char `model` throws `DataIntegrityViolationException` inside
  `CompletionPersistence.createPending`; the catch-all handler returns **500** instead of 400.
  Trivially reproducible client-triggered server error (alert noise; breaks §4.5 "invalid input → 4xx").

## MEDIUM

**R1 — security/auth**
1. **Login throttle keyed on composite (email|IP)** — `AuthThrottleService.kt:125-131` — a
   single IP spraying many emails is never throttled; also unbounded `LOGIN_FAILED` audit growth.
2. **Login timing oracle** — `AuthService.kt:68` — BCrypt short-circuits on unknown/disabled
   accounts (no decoy hash), so response timing reveals account existence.
3. **`GlobalExceptionHandler` catch-all maps 405/415/missing-param to 500** with ERROR-level
   stack traces — `GlobalExceptionHandler.kt:87-93`; it does not extend
   `ResponseEntityExceptionHandler`, so Spring's built-in 4xx mappings are lost. (Also the
   amplifier behind R4-H1 and R3-M1.)
4. **§4.5 default 256 KB request-body cap not found enforced anywhere** — only the import
   endpoint's 5 MiB bounded read exists. (Flagged "verify"; treat as unenforced until shown otherwise.)

**R2 — source-config core**
5. **Rule 27 `visibleWhen` cycle detection is unboundedly recursive** — `FilterRules.kt:360-371`
   — a ~20k-deep dependency chain overflows the stack; `StackOverflowError` escapes the
   `ValidationResult` contract.

**R3 — lifecycle/persistence/concurrency**
6. **`createSource` is check-then-act with no lock** — `SourceAdminService.kt:73`
   (`existsByApi` → `create`) — duplicate-api race (admin vs admin, or admin vs running bundled
   import) escapes as an unmapped DB unique-violation → 500 instead of the documented 409; in the
   import-race variant the whole import rolls back with a 500.
7. **Import force-publishes existing draft sources** — `BundledImportService.kt:156-172` — a
   draft's `publishedChecksum` is null so `identical` is never true; a routine re-import silently
   flips admin WIP `draft → active` into the served document. If the stanza said
   `lifecycle:"disabled"` the source still comes up ACTIVE (conflict-only), while the same stanza
   on an absent source would be created DISABLED — divergent outcomes.
8. **Created sources take raw payload index as `position`** — `BundledImportService.kt:143` —
   only equals payload order if existing catalog positions already match payload indexes; a new
   stanza can serve before an earlier-ordered existing source or collide with an unrelated
   position (api-string tiebreak), diverging from PLAN §12.2/§5 "payload order".
9. **Admin `listSources` N+1** — `SourceAdminService.kt:136-141` — two extra queries per head
   (published revision number + latest revision number): ~91 queries per request at the real
   ~45-source catalog size.

**R4 — completion / public API**
10. **Executor queue-wait billed against provider timeout; interrupt-immune providers wedge the
    pool** — `CompletionService.kt:165-175,248` — `future.get(timeout)` starts at submit while
    the hardcoded 8-thread pool has an unbounded queue: under >8 concurrent slow requests,
    request #9+ records `PROVIDER_TIMEOUT` for a call that never started; a provider that
    ignores interrupts can permanently occupy all 8 threads → every completion times out until
    restart.
11. **`Content-Type: application/json; charset=UTF-8` (normative §4.5) missing on all
    Jackson-rendered endpoints** — only `DocumentResponseWriter`, `/document/meta`, and
    `/sources/{api}` set it explicitly; `GET /sources`, completions, auth/admin emit bare
    `application/json`. Functionally benign (RFC 8259) but a normative-contract deviation.

## Verified sound (per slice)

**R1:** HS256 alg-confusion blocked; iss/aud/exp/nbf validators present; DB-backed per-request
revocation (enabled check + DB-derived authorities); last-admin guard under PESSIMISTIC_WRITE;
72-byte BCrypt password policy; client-IP resolver defaults to remoteAddr (no header trust);
actuator health-only.

**R2:** kcj-1 code-point key sort correct including surrogates; checksum over exact canonical
UTF-8 bytes; STRICT vs COMPATIBILITY parser modes correct; config model matches PLAN §7
field-for-field; no mutable validator state; image-strategy whitelist empty as specified.

**R3:** Uniform lock ordering (global `document_publication_state` FOR UPDATE first, then source
rows — no deadlock cycle); 10-step §9 publication sequence exact (validation under lock, one
truncated Clock instant shared, `nextval` under lock, snapshot before pointer move, `materialize`
`Propagation.MANDATORY`); supersede-then-publish keeps `uq_one_published_per_source` valid at
every statement; publishable-revision rules match §9 including 409 precedence; rollback is a true
forward-roll (new max+1 revision, byte-identical content, history immutable via `updatable=false`);
LifecycleStateMachine matches §9 (retire only from disabled, remove only from retired, removed
terminal, un-retire generic-gated); import 422 computed before any lock/write, exactly one
snapshot, no-op creates nothing, single `@Transactional` body → partial application impossible;
startup validators' boundary conditions correct (empty DB passes; misconfig fails with runbook
text, never auto-repairs); V2–V4 DDL faithful to §5 (composite FK makes cross-source pointers
unrepresentable, RESTRICT everywhere, `CHECK (id=1)` singleton, sequence START 100 NO CYCLE);
`@Modifying(flushAutomatically, clearAutomatically)` makes JPA/JDBC mixing safe.

**R4:** Three-transaction completion pattern real (no tx spans the provider call; separate
`CompletionPersistence` bean → no self-invocation); executor lifecycle clean (`@PreDestroy
shutdownNow()`, daemon threads, graceful drain); timeout→cancel interrupts the worker and the
interrupted worker performs no DB writes; §10 error-code catalog mapped exactly
(`RESULT_TOO_LARGE` unused is plan-conformant — truncate-to-SUCCEEDED); client error is always
the single generic constant, provider detail server-log-only and sanitized; prompt cap → 413,
blank → 400, page bounds enforced; non-owner GET → 404 (no id probing); orphaned RUNNING rows on
crash documented as harmless in §10 (no sweep required); V5 DDL matches entities with the one
normative index; `DocumentResponseWriter` — strong quoted ETag, `*`/comma-lists, `W/` never
matches, bodiless 304 with full headers, raw stored bytes (no converter), correct Content-Length;
`SourceQueryService` resolves via the authoritative pointer, lifecycle absence → active via model
default, REMOVED→410 before any document; build matches §3 exactly (Boot 3.5.16 / Kotlin 2.1.21 /
Java 21), dependency scopes clean; application yml prod-safe (health-only actuator,
`ddl-auto: validate`, `open-in-view: false`, no prod secret defaults, registration off in prod,
Swagger dev-only/ADMIN); `FlywayIncrementalOrderIT` genuinely uses fresh throwaway databases.

## Test-quality assessments

- **Concurrency ITs (R3):** real races — `CyclicBarrier` + thread pools driving the Spring proxy
  on separate transactions; `ConcurrentSameSourceRevisionIT` (8 threads) would reliably fail
  without the source-row lock. Caveats: publish races run a single 2-thread trial (probabilistic
  detection if the global lock were removed); no race coverage for import or duplicate-create.
- **Completion ITs (R4):** strong (authz, taxonomy via controllable provider, sanitization
  asserted against response and stored row). Gaps: `PROVIDER_UNAVAILABLE` /
  `INVALID_PROVIDER_RESPONSE` mappings and the truncation policy are untested.
- **ETag ITs (R4):** near-exhaustive conditional-GET semantics; served==stored byte-identity is
  hash-proven only on the admin snapshot route — the public `/document` route's guarantee is
  established indirectly.

## Suggested fix order (owner decision — nothing applied)

1. **R2-H1** header-name trim + token validation (security: credential can reach the public document).
2. **R4-H1** length-validate `model` (≤128) at the controller → 400.
3. **R1-M3** make `GlobalExceptionHandler` extend `ResponseEntityExceptionHandler` + map
   `DataIntegrityViolationException` → fixes the 405/415/param 500s and downgrades R3-M6/R4-H1's
   blast radius.
4. **R1-M4** enforce the §4.5 256 KB default body cap.
5. **R1-M1 + R1-M2** login throttle (add per-IP dimension) + decoy BCrypt hash.
6. **R3-M7/M8** import draft/position semantics (decide intended behavior, then pin with tests).
7. **R4-M10** bound the completion queue / start the timeout clock at execution.
8. The LOWs are cleanup-grade; batch at leisure.

## Review conduct (for the record)

Sequential Fable agents with a self-throttling token guard (`.agent/fable-guard.py`, 40k
input-tokens/rolling-minute budget vs the ~100k/min org hard limit; ledger + log in `.agent/`).
R1 died twice and R3 once to org-level 429s before the survival protocol (guard-first, ≤3 calls
per checkpoint, ~2,500-line read cap, single sequential tool calls) made every subsequent run
complete. Reviews were scoped to be disjoint; R2's validation-internals findings excluded import
orchestration, which R3 covered.
